# Plan for corepost-mobile-android

## Роль репозитория

Android panic-клиент для мгновенной блокировки устройства и подтверждения статуса через сервер CorePost.

## Текущее состояние

- Проект не в релизном состоянии:
  - `gradlew` не исполняемый;
  - `gradle.properties` содержит Windows-only `org.gradle.java.home`;
  - `AndroidManifest.xml` синтаксически сломан;
  - одновременно лежат `app/build.gradle` и `app/build.gradle.kts`;
  - Kotlin DSL не содержит `OkHttp`, хотя код его импортирует;
  - хранение настроек и секретов небезопасное;
  - релизная подпись и pipeline отсутствуют.
- По логике безопасности приложение сейчас подписывает запросы `emergencyId`, что надо согласовать с серверным контрактом.

## Что сделать

- Нормализовать сборочную систему:
  - оставить один Gradle DSL;
  - починить wrapper/permissions;
  - убрать платформенно-зависимые настройки;
  - добиться воспроизводимой сборки на текущем macOS-host.
- Исправить manifest, ресурсы и package structure.
- Пересобрать сетевой слой:
  - согласовать модель секрета с сервером;
  - убрать чувствительные данные из логов;
  - добавить обработку TLS, сертификатов и, при необходимости, pinning.
- Улучшить UX panic-flow:
  - onboarding/настройка;
  - статус устройства;
  - двухшаговое подтверждение lock;
  - ошибки соединения и понятные recovery paths.
- Перевести хранение конфигурации в безопасное хранилище (`EncryptedSharedPreferences`/Keystore).
- Добавить unit/instrumentation tests для HMAC, API client и UI state transitions.
- Подготовить релиз:
  - versioning;
  - signing config;
  - сборка release APK/AAB;
  - changelog;
  - публикация артефакта в GitHub Releases.

## Критерий готовности

- Приложение собирается в release на текущем macOS-host.
- Есть подписанный релизный артефакт и воспроизводимый процесс его сборки.
